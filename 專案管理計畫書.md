# 專案管理計畫書

## 1. 文件目的
本文件依 CMMI Level 3 精神，定義「升學與英語考題系統」之專案管理方法、生命週期、角色分工、工作產品、品質保證、風險管理、組態管理、溝通管理與驗收方式，作為專案執行與追蹤之基準。

## 2. 專案概述
- 專案名稱：升學與英語考題系統
- 專案性質：單體式 Node.js 線上考試平台
- 主要目標：
  - 提供題庫管理、試卷管理、學生作答、自動評分、人工批改與學習分析
  - 支援國小六年級、升國中資優班、國中七至九年級、教育會考與 GEPT 初級題型
  - 支援 AI 輔助出題與音訊/圖片上傳
- 系統範圍：
  - 後端：Express + SQLite
  - 前端：`public/` 靜態頁面
  - 資料庫：`exam.db`
  - AI 整合：OpenAI、Gemini、Claude

## 4. 專案生命週期
本專案採裁適後之瀑布式開發流程，並保留小幅迭代修正能力。

1. 啟始階段
   - 蒐集需求與利害關係人期望
   - 建立專案章程、範圍與里程碑
2. 需求分析階段
   - 建立系統分析規格書
   - 盤點角色、功能、資料與介面需求
3. 系統設計階段
   - 完成系統架構、資料庫、模組、API 與安全設計
4. 程式設計與建置階段
   - 實作後端、前端、資料種子與部署腳本
   - 執行單元與整合驗證
5. 測試與驗收階段
   - 執行手動測試、缺陷修正、驗收確認
6. 維運與持續改善階段
   - 執行缺陷分析、題目品質分析、版本控管與文件更新

## 8. 估算與排程管理
- 估算方式：
  - 功能點拆解
  - 歷史經驗比對
  - 關鍵模組採三點估算
- 排程基準：
  - 功能需求凍結後進入設計
  - 設計基準版通過後方可大量開發
  - 所有上線項目需完成手動驗證
- 追蹤方式：
  - 每週進度會議
  - 每週更新里程碑達成率
  - 針對風險與缺陷進行紅黃綠燈控管

## 9. 組態管理計畫
- 組態項目：
  - 原始碼：`server.js`、`database.js`、`llm.js`、`generate-public.js`
  - 前端頁面：`public/*.html`
  - 種子資料：`seed*.js`
  - 文件：各類 `.md`
  - 執行設定：`.env.example`、`package.json`
- 管理原則：
  - 使用 Git 進行版本控管
  - 主幹版本需保持可啟動
  - 不直接刪除正式資料庫 `exam.db`
  - 重要 schema 變更需先備份資料庫
- 版控基線：
  - 需求基線
  - 設計基線
  - 發版基線

## 10. 品質保證計畫
- 遵循 CMMI Level 3 的定義化流程，建立可重複執行之開發與審查步驟
- 品質活動：
  - 需求審查
  - 設計審查
  - 程式碼走查
  - 手動功能測試
  - 資安檢核
  - 文件一致性檢查
- 品質準則：
  - 啟動成功
  - 題庫 CRUD 正常
  - 試卷 CRUD 正常
  - 學生作答與成績查詢正常
  - 人工批改可回寫總分
  - AI 出題於金鑰存在時可正常運作

## 11. 風險管理計畫
| 風險編號 | 風險說明 | 影響 | 機率 | 因應策略 |
| --- | --- | --- | --- | --- |
| R1 | SQLite 單機資料庫容量或鎖定限制 | 中 | 中 | 定期備份、控管併發、保留升級 DBMS 方案 |
| R2 | LLM API 金鑰未設定或供應商異常 | 中 | 高 | 以預覽模式處理錯誤，保留人工建題流程 |
| R3 | 題目答案外洩 | 高 | 中 | 學生取題 API 隱藏答案，管理 API 以 `x-api-key` 保護 |
| R4 | 檔案上傳濫用或格式異常 | 中 | 中 | 限制 MIME、檔案大小與管理權限 |
| R5 | 資料庫 migration 失敗 | 高 | 低 | 先備份 `exam.db`，以交易與檢查欄位方式執行 |
| R6 | 人工批改流程未完成導致成績不完整 | 中 | 中 | 以 `pending/graded` 狀態控管並提供待批改清單 |
| R7 | 無自動化測試造成回歸風險 | 中 | 中 | 建立手動測試清單與版本發佈前驗證 |

## 12. 決策與問題管理
- 重大需求變更需經 PM、SA、開發代表共同評估
- 所有缺陷與議題需記錄：
  - 問題描述
  - 影響範圍
  - 優先級
  - 指派人員
  - 預計完成日
- 涉及資料結構變更之議題需同步更新分析、設計與程式文件

## 13. 溝通管理計畫
| 溝通項目 | 對象 | 頻率 | 內容 |
| --- | --- | --- | --- |
| 專案週會 | PM、SA、DEV、QA | 每週 | 進度、風險、議題 |
| 規格審查會 | PM、SA、SD、SME | 里程碑前 | 需求/設計審查 |
| 缺陷追蹤會議 | DEV、QA | 視需要 | 缺陷確認與修復狀態 |
| 驗收會議 | PM、使用單位 | 階段完成時 | 功能展示與驗收 |

## 14. 度量與分析計畫
- 進度度量：
  - WBS 完成率
  - 里程碑準時率
- 品質度量：
  - 缺陷數
  - 缺陷修復時間
  - 測試通過率
- 產品度量：
  - 題目總數
  - 作答次數
  - 各題正答率、錯答率、不會率
  - 題目鑑別度與難度偏差

## 15. 驗收管理
- 驗收範圍：
  - 啟動與部署
  - 題庫管理
  - 試卷管理
  - 學生作答與交卷
  - 成績分析
  - 人工批改
  - AI 出題與媒體上傳
- 驗收準則：
  - 各頁面可正常操作
  - API 回應符合規格
  - 資料可正確寫入 SQLite
  - 權限與安全控管符合設計
  - 文件與實作一致

## 17. 裁適說明
本專案為中小型單體系統，CMMI Level 3 採「文件化、標準化、可追蹤」之實務裁適，不強制導入大型組織的完整流程系統，但保留以下必要元素：
- 有定義之生命週期
- 有角色責任分工
- 有文件基線與變更控管
- 有品質保證與風險管理
- 有審查、驗收與持續改善機制
