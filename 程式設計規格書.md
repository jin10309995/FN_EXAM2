本文件說明目前程式實作結構、主要函式、資料處理邏輯、例外處理方式與維護規範，作為程式設計、修改與交接依據。

## 3. 程式設計原則
- 採 CommonJS 模組
- API 以 JSON 為主要交換格式
- 直接使用 SQL 操作 SQLite
- 核心多步驟寫入以 transaction 保護
- 例外集中於全域錯誤中介層處理

## 4. 模組詳細設計
### 4.1 `server.js`
#### 4.1.1 啟動與中介層
- 載入 `.env`
- 建立 `Express` 實例
- 建立 `uploads/audio`、`uploads/images`
- 設定 `helmet`、`cors`、`express.json`
- 設定 API 與提交速率限制
- 設定靜態資源路由

#### 4.1.2 管理權限函式
`requireAdmin(req, res, next)`
- 讀取 `process.env.ADMIN_API_KEY`
- 驗證 `x-api-key`
- 驗證失敗回傳 401
- 若未設定管理金鑰則視為開發模式放行

#### 4.1.3 題目 API
- `GET /api/questions`
  - 依條件組裝 SQL `WHERE`
  - 支援分頁
- `POST /api/questions`
  - 驗證題型、學段與必要欄位
  - 新增單筆題目
- `PUT /api/questions/:id`
  - 以 `COALESCE` 支援部分欄位更新
- `DELETE /api/questions/:id`
  - 刪除指定題目
- `POST /api/questions/batch`
  - 使用 transaction 批次新增題目
- `GET /api/questions/random`
  - 支援一般隨機或加權隨機

#### 4.1.4 試卷 API
- `GET /api/exams`
  - 僅管理者可查
  - 回傳題數與總分彙總
- `GET /api/exams/:id`
  - 含答案的完整試卷，僅管理者可用
- `GET /api/exams/:id/take`
  - 學生作答用，不回傳答案
- `POST /api/exams`
  - 建立試卷與題目關聯
- `PUT /api/exams/:id`
  - 更新試卷與關聯題目
- `DELETE /api/exams/:id`
  - 清除 submission 與 answer detail 後刪除試卷

#### 4.1.5 作答評分 API
`POST /api/exams/:id/submit`
- 輸入：`student_name`、`student_id`、`answers`
- 邏輯：
  - 驗證試卷存在且為 `active`
  - 讀取試卷題目與正解
  - 依題型產生 `details`
  - 在 transaction 中寫入 submission 與明細
  - 更新題目統計欄位
  - 回傳成績百分比

#### 4.1.6 題目封存函式
`archiveAndReplace()`
- 查詢答對超過 5 次且未封存題目
- 封存原題
- 呼叫 `generateQuestions()` 產生難度較高替換題
- 寫入 `questions`

#### 4.1.7 分析與統計 API
- `GET /api/submissions/:id`
  - 顯示單次作答詳細成績
- `GET /api/submissions/:id/analysis`
  - 建立科目、難度、弱點與建議分析
- `GET /api/exams/:id/stats`
  - 提供試卷作答統計與最常錯題
- `GET /api/analytics/difficulty-calibration`
  - 標示難度與實際難度偏差
- `GET /api/analytics/question-quality`
  - 計算通過率、鑑別度與品質旗標
- `GET /api/analytics/student-ability`
  - 依歷史作答估算學生能力
- `GET /api/recommendations`
  - 根據歷史弱點推薦題目

#### 4.1.8 人工批改 API
- `GET /api/exams/:id/pending-grading`
  - 查詢待批改題目
- `PUT /api/answer-details/:id/grade`
  - 驗證 `rubric_score`
  - 限制分數不超過題目滿分
  - 更新狀態為 `graded`
  - 重算 submission 分數

#### 4.1.9 上傳 API
- 音訊：`POST /api/audio/upload`
- 圖片：`POST /api/image/upload`
- 皆以 `multer.diskStorage` 儲存到實體路徑

### 4.2 `database.js`
#### 4.2.1 連線初始化
- 使用 `better-sqlite3`
- 啟用 `WAL` 與 `foreign_keys`

#### 4.2.2 Schema 建立
- 建立六個核心資料表
- 初始化主鍵與外鍵
- 建立必要 `CHECK` 條件

#### 4.2.3 Migration 設計
- 透過 `PRAGMA table_info` 檢查欄位存在與否
- 缺少欄位時執行 `ALTER TABLE`
- 題型約束變更時採 rename-copy-drop 流程重建 `questions`
- 針對外鍵被誤改寫情境，重建 `exam_questions` 與 `answer_details`
- 最後補入預設 `subjects`

### 4.3 `llm.js`
#### 4.3.1 主要函式
`generateQuestions(provider, userPrompt)`
- 依 provider 分派到：
  - `callOpenAI`
  - `callGemini`
  - `callClaude`

#### 4.3.2 回應解析
`parseJsonResponse(raw)`
- 先直接 `JSON.parse`
- 失敗時嘗試擷取陣列 JSON 區段
- 若回傳物件包陣列，抓取第一個陣列值

### 4.4 `generate-public.js`
- 以字串樣板產生前端頁面
- 用於重新建構 `public/`
- 所有頁面為靜態 HTML，前端透過 `fetch` 直接呼叫 API

## 6. 演算法與邏輯說明
### 6.1 加權隨機選題
- 使用 `wrong_count + dont_know_count * 2 + 1` 作為權重基礎
- 不會的題目優先度高於答錯

### 6.2 Rasch 能力估計
- 將難度 1 到 5 映射到 logit
- 採迭代法求解能力值
- 最終回傳 1.0 到 5.0 的顯示值

### 6.3 題目品質
- 通過率過高或過低標記異常
- 鑑別度以高低分組通過率差異計算
- 綜合產生 `quality_flags` 與 `quality_score`

## 7. 例外與錯誤處理
- 所有路由應回傳 JSON 錯誤訊息
- 預期錯誤於路由中處理：
  - 欄位缺失
  - 找不到資料
  - 權限不足
  - 狀態不符
- 非預期錯誤由全域錯誤處理器攔截

## 8. 安全程式設計要求
- 禁止學生端 API 回傳答案欄位
- 禁止未授權存取管理 API
- 上傳檔案需驗證型別與大小
- SQL 一律使用參數化查詢
- 避免將內部錯誤明細直接回給前端

## 9. 維護與擴充規範
- 新增欄位時同步更新：
  - `database.js`
  - 相關 API 驗證與 SQL
  - 分析或前端顯示邏輯
  - 規格文件
- 新增題型時同步更新：
  - 題型白名單
  - 作答頁渲染
  - 評分邏輯
  - 分析報表

## 10. 測試建議
- 啟動測試：`node server.js`
- 功能測試：
  - 題庫 CRUD
  - 試卷 CRUD
  - 交卷與成績
  - 待批改與人工給分
  - 分析與推薦
  - 媒體上傳
  - AI 產題
- 若後續新增自動化測試，建議建立獨立測試資料庫並採 `*.test.js`

## 11. 已知設計限制
- 商業邏輯集中於單一 `server.js`
- 缺少 service/repository 分層
- 目前無正式單元測試與 CI
- 無登入系統，管理權限以單一金鑰控管
